## 1. Purpose

### Why This Feature Exists

The current prototype uses basic colored squares with letters for unit identification. While functional for testing core mechanics, these placeholders don't convey character personality, class fantasy, or tactical role at a glance. Playtesters need more visual information to connect with the mercenary company fantasy and make tactical decisions. This feature replaces generic placeholders with distinctive programmatically-generated sprites that convey character identity and ability function.

### What It Enables

- Players instantly recognize each party member by silhouette and visual style
- Ability icons clearly communicate function before reading tooltips
- Visual distinction between character classes (Fighter vs Rogue vs Cleric)
- Soldiers and enemies have appropriate faction-specific appearances
- Status effects are represented with recognizable icons
- The "Build & Command" fantasy is reinforced through visible character identity

### Success Criteria

| Criterion | Measurement |
|-----------|-------------|
| Character recognition | Playtesters identify units without reading names |
| Ability clarity | Icons suggest function without tooltip |
| Class distinction | Fighter/Rogue/Cleric visually obvious |
| Faction identification | Party/Soldier/Enemy instantly distinguishable |
| Visual consistency | All assets share cohesive art style |

## 12. Implementation Notes

### Approach Rationale

**Why programmatic generation instead of PNG files?**
- Flexibility to adjust colors and sizes without re-exporting
- Single source of truth for all visual parameters
- Easy to maintain consistency across sprites
- Smaller repository size (no binary assets)
- Enables dynamic effects (glow, pulsing, etc.)

**Why these specific visual designs?**
- Each character has distinctive silhouette readable at small sizes
- Class roles immediately apparent (tank=big, rogue=small, cleric=robes)
- Faction colors (blue party, green soldier, red enemy) established
- Ability icons suggest function through universal symbols

### Gotchas to Watch

1. **Anti-aliasing**: Pixel art should use nearest-neighbor scaling
2. **Glow effects**: May need separate CanvasItem for performance
3. **Color consistency**: Use constants, not hardcoded values
4. **Letter visibility**: Ensure contrast against all background colors
5. **Size constraints**: 48x48 leaves limited detail space

### Future Considerations

- **Animation frames**: Each sprite could have idle/attack/hurt frames
- **Equipment variations**: Weapon/armor changes visible on sprite
- **Status effect overlays**: Visual changes when buffed/debuffed
- **Direction facing**: Sprites may need left/right variants
